# PersonalAgent 项目上下文

更新时间：2026-06-26

本文档是项目的长期上下文账本，用于维护“下一步要做什么、已经做完什么、哪些方案已经证实不适合继续走、项目沉淀了哪些经验”。新会话应先读 `PROJECT_HANDOFF.md`、`AGENTS.md`、本文档，再读具体设计方案和代码入口。

## 维护规则

- 每完成一次方案落地、阶段验收或重要修复，都必须同步更新本文档。
- 更新内容至少覆盖：计划开发任务、已完成任务、已放弃或证实不可用的方案、经验教训、验证结果。
- 如果某次落地有独立完成情况文档，应在本文档中链接该文档，并摘要最终状态。
- 本文档记录工程事实和决策，不记录 `.env`、`.personal_agent/`、`.tmp/`、日志、密钥、本地数据库内容等运行态或敏感信息。
- 若实现过程中发现原方案有错误，应同时修正方案文档或完成情况文档，并在本文档的“经验教训”中记录原因。

## 当前项目快照

- 项目定位：单用户、本地优先的智能助手。
- 后端：`personal_agent/`，FastAPI + SQLite。
- 前端：`frontend/`，React + Vite + TypeScript。
- 本地数据库：默认 `.personal_agent/agent.db`，属于运行态，不提交。
- 主要工程入口见 `PROJECT_HANDOFF.md` 和 `AGENTS.md`。
- 最近主线：开发任务编排层、当前草稿页、草稿与任务身份体系已经进入可用闭环，当前草稿页关系认知优化也已完成收口。
- 最近完整验收记录来自 `设计方案/当前草稿页关系认知优化完成情况.md`：`frontend` 下 `npm run typecheck` 和 `npm run build` 通过。

## 已证实不适合继续使用的方案

### 直接暴露 `task_uid` 作为用户任务编号

- 结论：不可用。
- 原因：`task_c363a...` 这类技术编号不直观，无法帮助用户理解任务线。
- 替代方案：后端派生 `display_code`，如 `T1/T2`；业务层继续使用 `task_uid`。

### 用 `current_revision` 表示草稿候选序号

- 结论：不可用。
- 原因：`current_revision` 是单个 draft 内部修订版本，不是同一任务阶段下多个候选草稿的序号。
- 替代方案：使用 `candidate_index` 和 `stage_candidate_count` 表达“候选 N/M”。

### 用 `is_active` 判断任务阶段当前采用候选

- 结论：不可用。
- 原因：`is_active` 当前是 session + document_type 作用域，不能准确表达某个 task 的某个阶段当前采用了哪个 draft。
- 替代方案：基于 `_effective_stages(task_uid)` 选中的 `stage.draft_uid` 计算 `is_stage_current_candidate`。

### 第一版就持久化 `display_code`

- 结论：暂不采用。
- 原因：当前展示编号可以从 `agent_tasks.id` 和 `session_uid` 稳定派生，立即迁移 DB 会增加成本和兼容风险。
- 替代方案：第一版只做 additive payload 字段；后续有跨导入/同步/删除后永久编号需求时再评估持久化。

### 把未关联草稿按会话拆成多个“未关联草稿”组

- 结论：不可用。
- 原因：无 `task_uid` 的草稿没有任务线语义，按会话拆分会制造新的“伪任务”。
- 替代方案：统一进入单一“未关联草稿”组；如后续需要细分，单独设计草稿维度的信息架构。

### 无 active task 时用“继续”等关键词启动新任务

- 结论：不可用。
- 原因：用户的“继续”可能是普通对话语义，本地关键词会误触发开发任务。
- 替代方案：继续保持普通聊天意图由 LLM router 判断，不引入本地关键词任务路由。

### 新会话继承旧任务材料

- 结论：不可用。
- 原因：会造成上下文泄漏，让新任务混入旧材料。
- 替代方案：材料、任务筛选、pending/error 状态必须按 session 隔离。

### 绕过 candidate draft 策略直接应用真实代码 patch

- 结论：禁止。
- 原因：项目的代码 patch 能力要求默认产出 candidate draft，便于审查、验证和回滚。
- 替代方案：继续通过受控候选、验证摘要和用户确认路径推进真实变更。

## 经验教训

- 用户可理解的身份体系应按“任务 -> 文档阶段 -> 候选 -> 修订版本 -> 当前采用”建模，不能把数据库主键直接暴露成产品语言。
- 展示字段优先后端派生，前端负责组织呈现；这样可以避免每个前端入口各自猜业务语义。
- additive payload 字段适合渐进演进；已有字段语义不清时，不要复用旧字段表达新概念。
- `display_code`、`candidate_index` 这类展示编号不能替代 `task_uid`、`draft_uid` 等真实业务主键。
- 会话切换、新建会话、全部模式是前端状态最容易串线的三条路径；新增筛选或联动时必须逐一检查。
- “未关联”不是一种任务状态；没有 `task_uid` 时不要强行套用任务候选语义。
- “当前采用候选”只表示任务阶段当前选中的候选，不等于质量通过、已完成或可交付；质量状态必须独立展示。
- “会话当前草稿”和“任务阶段当前采用候选”是两套语义，前端按钮和标签必须分开命名。
- “未关联草稿”是草稿维度，不是任务维度；命名、分组和说明都不能再借用“未关联任务”这种误导性说法。
- 当前草稿页的“样式收尾”不能顺手扩成新交互能力；像抽屉拖拽调宽这类行为变更应单独立项和评审。
- 草稿删除默认应采用软删除，不能第一版就物理删除 revisions；任务候选、质量失败追踪和版本历史都需要可恢复的审计上下文。
- 删除当前采用候选的影响提示必须以后端实时重算结果为准；前端只能展示 impact，不能缓存或复写候选关系判断。
- 回收站模式应与普通草稿模式严格隔离，激活、修订、重新生成、打开/下载等动作要集中收敛，避免 UI 上出现“已删除但还能继续工作流”的错觉。
- 质量失败、阻塞、修复候选、学习候选都应默认走 candidate/approval 机制，不直接沉淀为 approved 事实。
- 单用户本地工具不需要过度设计高并发架构，但 SQLite 长耗时写入和前端轮询会产生真实锁冲突风险，WAL 和 `busy_timeout` 仍有价值。
- 大模块存在但不宜为拆而拆；优先在触碰具体职责时顺手抽离，降低一次性重构风险。
- 文档落地后必须有完成情况文档或上下文摘要，否则新会话只能靠 git log 和散落方案猜当前状态。
